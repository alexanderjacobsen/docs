Tools are the heart of Mantl. They turn the chat from "Q&A" into "an assistant that can actually navigate, search, mutate, and read state." A tool is a function the AI decides to call when it judges it useful for the user's request.

## Anatomy of a tool

```ts
import { z } from 'zod';
import { ClientTool } from '@getmantl/angular';

const navigateTo: ClientTool = {
  description: "Navigate the user to a page in the app",
  parameters: z.object({
    path: z.string().describe("The path to navigate to, e.g. /orders/42"),
  }),
  execute: async ({ path }) => {
    await this.router.navigateByUrl(path);
    return { success: true };
  },
};
```

Three parts:

- **`description`** — natural-language explanation of what the tool does. The model uses this to decide when to call it. Be specific.
- **`parameters`** — a [Zod](https://zod.dev) schema or plain JSON Schema describing the tool's input. With Zod you get TypeScript inference inside `execute` for free, plus runtime validation before your code runs.
- **`execute`** — the function that actually runs in the browser when the AI calls the tool. Receives the validated input and returns whatever the AI should see.

## Registering tools

Two ways:

### Globally on `<mantl-chat>`

For tools available everywhere in your app:

```html
<mantl-chat tokenUrl="/api/token" [tools]="globalTools" />
```

```ts
globalTools: ClientTools = {
  navigateTo,
  searchProducts,
  getCurrentUser,
};
```

### Locally via `[mantlTools]`

For tools that depend on a specific page's state, register them on the page's host element via the directive. They're added when the element initializes and removed when it's destroyed.

```html
<section [mantlTools]="productTools">
  <!-- product page -->
</section>
```

See [`[mantlTools]`](/angular/mantl-tools) for the full reference.

The two sets are merged. If the user is on the products page, the AI sees both global tools and the local ones.

## TypeScript and input typing

The `ClientTool` interface uses `unknown` for the input parameter to stay schema-agnostic. Inside `execute`, cast to the shape implied by your `parameters`:

```ts
execute: (input: unknown) => {
  const { path } = input as { path: string };
  // ...
}
```

The model validates against your `parameters` schema before calling, so by the time `execute` runs the input matches the shape. The cast is just for TypeScript.

If you prefer compile-time typing, use a tagged shape:

```ts
type NavInput = { path: string };
const navigateTo: ClientTool<NavInput> = {
  description: "...",
  parameters: { /* JSON Schema */ },
  execute: ({ path }) => { /* typed */ },
};
```

## Returning data from a tool

Whatever your `execute` returns becomes the model's view of the result. Return structured data — objects, arrays, JSON — when you want the model to reason about it:

```ts
const getOrders: ClientTool = {
  description: "Get all orders for the current user",
  parameters: { type: "object", properties: {}, required: [] },
  execute: async () => {
    const orders = await this.http.get<Order[]>('/api/orders').toPromise();
    return orders;
  },
};
```

The model sees the JSON and can summarize, filter, count, etc.

If your tool just performs an action (navigate, mutate, dismiss), return a small acknowledgement so the model knows it succeeded:

```ts
return { success: true };
```

## Errors

If your `execute` throws, Mantl catches it and reports the error to the model as a regular result, so the model can react ("Sorry, I couldn't fetch your orders — would you like me to try again?"). You don't need a try/catch unless you want a custom error message:

```ts
execute: async (input: unknown) => {
  const { id } = input as { id: string };
  try {
    return await this.deleteOrder(id);
  } catch (err) {
    return { success: false, message: "Order not found" };
  }
}
```

## Confirmation flow

Some tools should ask the user before running — destructive actions, payments, anything sensitive. Add `requiresConfirmation: true`:

```ts
const deleteOrder: ClientTool = {
  description: "Delete an order by ID",
  parameters: {
    type: "object",
    properties: { id: { type: "string" } },
    required: ["id"],
  },
  requiresConfirmation: true,
  execute: async (input: unknown) => {
    const { id } = input as { id: string };
    await this.http.delete(`/api/orders/${id}`).toPromise();
    return { deleted: id };
  },
};
```

The user will see an "Approve / Deny" prompt in the chat before the tool runs.

## Customizing the tool call display

Tool calls render inline in the chat as a small "tool chip" — an icon, a label, and a status line. Two fields on a tool let you tailor that:

| Field | Type | What it changes |
| --- | --- | --- |
| `displayName` | `string` | The label on the chip. Defaults to a humanized form of the tool key (`openOrder` → "open order"). |
| `formatOutput` | `(output) => string` | The one-line status shown after the tool runs, in place of the default `"<name> completed"`. |

```ts
productTools: ClientTools = {
  searchProducts: {
    description: "Search the catalog",
    parameters: z.object({ query: z.string() }),
    displayName: "Search products",
    formatOutput: (output) => `Found ${(output as { results: unknown[] }).results.length} matches`,
    execute: async ({ query }) => {
      const results = await this.http.get(`/api/products?q=${query}`).toPromise();
      return { results };
    },
  },
};
```

### Rendering details with an Angular component

When a tool finishes, you can show a richer card below the chip that visualises its input + output. `AngularClientTool` exposes two fields for this:

| Field | Type | What it does |
| --- | --- | --- |
| `detailsComponent` | `Type<DetailsComponent>` | An Angular component class with `@Input() input` and `@Input() output`. Rendered in an expandable panel below the chip. |
| `defaultExpanded` | `boolean` | Open the panel by default instead of waiting for a click. Defaults to `false`. |

The component is instantiated with Angular's `createComponent` using the host app's `EnvironmentInjector`, so DI, services, signals, `RouterLink`, theming — everything you'd expect — work exactly as in any other Angular component in your app. There's no React anywhere in the API you write.

```ts order-details.component.ts
import { Component, Input } from '@angular/core';

interface OrderInput { id: string; }
interface OrderOutput {
  id: string;
  customer: string;
  total: number;
  items: Array<{ name: string; qty: number }>;
}

@Component({
  selector: 'app-order-details',
  standalone: true,
  template: `
    <div class="order-card">
      <strong>Order {{ output?.id }}</strong>
      <span>{{ output?.customer }}</span>
      <ul>
        @for (item of output?.items ?? []; track item.name) {
          <li>{{ item.name }} × {{ item.qty }}</li>
        }
      </ul>
      <div>Total: \${{ output?.total }}</div>
    </div>
  `,
})
export class OrderDetailsComponent {
  @Input() input?: OrderInput;
  @Input() output?: OrderOutput;
}
```

```ts orders.component.ts
import { AngularClientTools } from '@getmantl/angular';
import { OrderDetailsComponent } from './order-details.component';

ordersTools: AngularClientTools = {
  openOrder: {
    description: "Open an order by id",
    parameters: z.object({ id: z.string() }),
    execute: async ({ id }) => this.orders.getOrder(id),
    detailsComponent: OrderDetailsComponent,
    defaultExpanded: true,
  },
};
```

Use `AngularClientTools` (not `ClientTools`) as the tool record's type when any entry uses `detailsComponent` — that's the type that knows about the field.

A few notes on how the mount works:

- **Inputs are set once at mount.** When the tool reaches the `output-available` state, `input` and `output` are stable, so they're passed via `setInput` exactly once. Components don't get update notifications mid-stream.
- **Lifecycle is normal.** `ngOnInit`, `ngOnDestroy`, `OnPush`, signals, animations all behave as anywhere else. The component is destroyed via `componentRef.destroy()` when the chat is closed or the conversation is cleared.
- **`@Output()` events don't go anywhere.** The chat doesn't subscribe to outputs from your details component — treat it as a pure display surface.

<Note>
For very advanced cases, the underlying `ClientTool` also accepts a raw `details: (input, output) => unknown` callback. `@getmantl/angular` wraps `detailsComponent` into that internally; you almost never need to use it directly.
</Note>

## Schema choices

Mantl accepts both Zod schemas and plain JSON Schema for `parameters`.

<CodeGroup>
```ts Zod
import { z } from 'zod';

parameters: z.object({
  query: z.string().describe("Search query"),
  limit: z.number().int().min(1).max(100).default(10),
})
```

```ts JSON Schema
parameters: {
  type: "object",
  properties: {
    query: { type: "string", description: "Search query" },
    limit: { type: "integer", minimum: 1, maximum: 100, default: 10 },
  },
  required: ["query"],
}
```
</CodeGroup>

Zod is recommended — you get TypeScript inference on the `execute` argument for free (no manual `input as { … }` cast) and runtime validation before your code runs. Just install it (`npm install zod`); the rest is automatic. JSON Schema is fine if you'd rather not add the dep.

## Tips for good tools

- **Be specific in descriptions.** "Get user's data" is bad; "Fetch the current user's recent orders, last 30 days" is good.
- **Use `description` on every parameter.** It's how the AI knows what each field means.
- **Return small, focused payloads.** Don't dump 500 rows when 10 will do — the model will use them all in context.
- **Make tools idempotent where possible.** The AI may retry. A `createOrder` that creates duplicates on retry is bad; one that requires an idempotency key is good.
- **Pull dependencies from DI, not from closures.** Injecting `Router` / `HttpClient` / your own services into the component that owns the tool lets `execute` use them naturally and keeps tools testable.
